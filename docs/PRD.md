# LaunchAI 产品需求文档(PRD)

> **版本**:v0.1(planning)
> **状态**:架构已拍板,待实现
> **配套文档**:`COMPETITIVE_ANALYSIS.md` / `TECH.md`

---

## 1. 产品定位

### 1.1 一句话

**LaunchAI 是 Chrome 扩展开发者的自主化 AI Agent 营销助理**:输入一个 Chrome Web Store URL,系统自主完成竞品分析、文案生成、多渠道发布资产打包、发布计划制定。

### 1.2 核心理念

> **让 AI 做决策,人类做监督。**

| 原则 | 落地要求 |
|---|---|
| **输入即结果** | 用户只输入 URL,其余全自动 |
| **最大化 AI 自动化** | 抓取、分析、生成、推荐全自动,无表单填写 |
| **决策过程可视化** | 每一步 Agent 决策都展示给用户:做了什么、为什么、用了什么数据 |
| **闭环学习** | 用户微调 → 系统记忆 → 下次输出更优(v3) |
| **人工监督** | 发布动作必须人工确认,绝不代发(防 spam / 合规风险) |

### 1.3 产品愿景演进

| 阶段 | 定位 | 目标用户规模 | 时间窗 |
|---|---|---|---|
| **v1** | Chrome 扩展开发者 launch 副驾 | 全球 ~10 万 Chrome 扩展开发者 | 0–6 月 |
| **v2** | B2D(开发者工具)多渠道 launch 编排器 | 全球独立开发者 ~500 万 | 6–18 月 |
| **v3** | 闭环优化型 launch agent | 同上 + 监控/迭代付费层 | 18 月+ |

## 2. 目标用户

### 2.1 v1 主用户画像

- **身份**:Chrome 扩展独立开发者(独立或 1–3 人小团队)
- **典型产品**:生产力扩展、AI 工具扩展、screenshot 工具、SEO 助手等
- **痛点**:
  1. 不擅长写营销文案,尤其面向 Web Store 评测员
  2. 不知道发布到哪些 subreddit / Indie Hackers / HN
  3. 没时间做竞品 listing 对比
  4. PH launch 一天最多投入 2 小时,需要全套资产开箱即用
- **付费意愿**:高(已有 Chrome Store Listing 优化外包市场,$50–$500/次)
- **首批种子用户来源**:项目发起人自己(docMask.app)+ Twitter / Reddit `r/chrome_extensions`

### 2.2 反目标用户(v1 不服务)

- 大公司营销团队(决策链长、需要审批流)
- 非英文市场产品(v1 只做英文输出)
- 实体产品 / 服务业(不是 dev tool)

## 3. 范围(v1 Scope)

### 3.1 核心功能

#### F1:URL 输入与自主抓取

- **输入**:Chrome Web Store URL(主)或任意产品官网 URL(辅)
- **Agent 行为**:
  - 自动判断 URL 类型(CWS / 官网 / GitHub README)
  - 优先调用 chrome-stats 等公开数据源
  - 退化到 Playwright 抓取 DOM
  - 抓取目标产品 + Top 10 同类扩展
- **可视化**:抓取进度、数据源、抓取到的字段

#### F2:多 Agent 分析

- **Crawler Agent** → 原始数据
- **Analyst Agent** → 卖点 / 痛点 / 关键词 / 用户评论摘要 / 调性
- **Competitor Agent** → 竞品差异化点 / listing 长度分布 / 评分分布
- **可视化**:每个 agent 的输入、输出、置信度

#### F3:多渠道资产生成

| 渠道 | 输出 |
|---|---|
| **Chrome Web Store Listing** | 优化版 title / short description (132 字) / detailed description / promo tile 文案 / screenshot caption × 5 |
| **Product Hunt** | tagline (60 字) / description / first comment / maker comment / 媒体包说明 |
| **Reddit** | 3 个候选 subreddit + 每个的帖子草稿(标题 + 正文)+ 自我推广合规提示 |
| **Hacker News** | Show HN 标题 + 正文草稿 + 评论应对策略 |
| **X / Twitter** | Launch thread(5–7 条)+ Hook tweet + 转发话术 |
| **Indie Hackers** | Milestone 帖子草稿 |
| **发布日历** | 7 天发布时序建议(基于各平台最佳发布时间) |

每类资产生成 **3 个版本**,Critic Agent 推荐最优 + 给出选择理由。

#### F4:Critic Agent 推荐

- 评估每个版本的:**调性(B2D 友好度)** / **关键词密度** / **风险(spam / 自我推广过强)**
- 推荐"最优组合",给出可解释的选择理由
- 用户可以一键采纳或选择其他版本

#### F5:发布执行(轻量)

- **一键复制**:每个资产带独立复制按钮
- **深链跳转**:`reddit.com/submit?title=...&text=...` / `news.ycombinator.com/submit` / X intent URL
- **计划提醒**:用户设置发布时间,系统邮件/Webhook 提醒(可选)
- **❌ 不做**:服务端代发(避免 spam / 账号风险 / 合规风险)

#### F6:决策可视化 Dashboard

- 时间轴展示每个 Agent 的执行步骤
- 每步可点开看详情(prompt、tool call、输出)
- "为什么这么决定"的解释卡片
- 用户可以对任何一步**重跑** / **调参** / **手动覆盖**

#### F7:用户反馈闭环(v1 仅记录)

- 用户对每个版本的"采纳 / 修改 / 弃用"行为被记录
- v1 仅做 schema 预留 + 数据采集
- v2 用于 prompt 优化和个性化

### 3.2 v1 不做的(明确 out-of-scope)

- ❌ Chrome 浏览器扩展(v2 考虑)
- ❌ 服务端自动发布到任何平台
- ❌ Reddit 评论自动回复(避开 ReplyAgent 红海)
- ❌ Meta / Google 广告投放(避开 Pixero 赛道)
- ❌ 落地页生成(避开 Sitekick 赛道)
- ❌ 视频脚本生成(v2 集成 revid.ai)
- ❌ 多语言输出(v2)
- ❌ 团队协作 / 多用户(v2)

## 4. 用户故事

### Story 1:首次 launch 准备(主流程)

```
作为 Chrome 扩展独立开发者
我希望粘贴我的 Web Store URL 后 5 分钟内拿到完整 launch 资产包
这样我就可以在 PH launch 前一晚不用熬夜写文案

验收标准:
- 输入 URL 后无需任何额外填写
- 5 分钟内输出 6 类渠道资产
- 每类资产至少 3 个版本可选
- Dashboard 实时显示 agent 进度
```

### Story 2:看到 AI 的决策过程

```
作为对 AI 输出有疑虑的开发者
我希望理解 Critic 为什么推荐版本 B 而不是 A
这样我可以决定是否信任这个推荐

验收标准:
- 推荐卡片下方有"为什么推荐这个"的解释
- 解释包含具体的评分项(调性 / 关键词 / 风险)
- 我可以一键切换到其他版本而不丢失进度
```

### Story 3:迭代优化某个版本

```
作为对推荐版本不满意的开发者
我希望能告诉系统"这个版本太 markety 了,重新生成"
这样我能快速得到符合调性的版本

验收标准:
- 任何资产都有"重新生成 + 微调 prompt"按钮
- 可以指定调整方向("更技术向" / "更短" / "强调免费")
- 重新生成只重跑 Writer Agent,不重新抓取
```

### Story 4:发布日的"一键复制"

```
作为正在 PH launch 的开发者
我希望发布按钮一键就能让我跳到目标平台带好文案
这样我不需要在 5 个 tab 之间复制粘贴

验收标准:
- Reddit 按钮 → 跳到 r/chrome_extensions 的 submit 页且标题正文已填好
- HN 按钮 → 跳到 Show HN 提交页且正文已填
- 必要时,文案以剪贴板复制方式提供
```

### Story 5:发布后(v3)

```
作为已经 launch 完的开发者
我希望系统帮我监控 PH 排名 / Reddit 评论
并根据反馈给我"第二轮发什么"的建议
```

## 5. 非功能性需求

| 维度 | 目标 |
|---|---|
| **生成时间** | 单次完整流程 ≤ 5 分钟(包含抓取 + 分析 + 生成) |
| **抓取成功率** | CWS URL ≥ 95%,通用 URL ≥ 85% |
| **可视化延迟** | Agent 步骤更新 ≤ 2 秒 |
| **成本上限** | 单次免费用户运行 ≤ $0.30(LLM + 抓取),付费 ≤ $1.50 |
| **可解释性** | 每个 LLM 输出必须附带"输入数据 + prompt + 决策依据" |
| **数据隐私** | 用户 URL 和生成内容不用于训练公开模型 |

## 6. 营收模型

### 6.1 定价层(初步)

| 层级 | 价格 | 包含 | 目标用户 |
|---|---|---|---|
| **Free** | $0 | 1 次完整 launch / 月,基础模型(GPT-4o-mini) | 试用 |
| **Solo** | $19 / 月 | 3 次 launch / 月,优质模型(GPT-4o / Claude Sonnet),全部渠道 | 独立扩展开发者 |
| **Pro** | $49 / 月 | 10 次 launch / 月,定制 prompt,优先支持,API access | 多产品 maker |
| **Lifetime**(早期) | $99 一次 | Solo 等级 1 年 | PH launch 早鸟 |

### 6.2 单位经济估算

- 单次完整运行成本(LLM + 抓取):约 $0.40(详见 `TECH.md`)
- Solo $19/月 × 3 次 = $1.20 成本 → ~94% 毛利
- Pro $49/月 × 10 次 = $4.00 成本 → ~92% 毛利

### 6.3 付费转化关键

1. **首次免费体验必须惊艳**——5 分钟全自动产出,质量可见
2. **Critic 的"为什么推荐"是付费触点**——免费层只显示推荐结果,付费层展示完整推理
3. **多渠道资产是锁点**——免费层只输出 PH + CWS,付费层全渠道

## 7. 成功指标(Metrics)

### v1 北极星指标

**LDR(Launch Delivery Rate)**:用户输入 URL 后 7 天内**真实在某个渠道发布**的比例

> 不是简单"生成了文案",而是"用户真的发了"。这是 PMF 信号。

### 配套指标

| 指标 | v1 目标(3 个月内) |
|---|---|
| 月活用户(MAU) | 1,000 |
| 完成首次生成的用户 | 800(80%) |
| 7 天内发布的用户(LDR) | 240(30% × MAU) |
| 付费转化率 | 5%(50 付费 / 1,000 MAU) |
| 月度 ARR | $1,000(50 × $19 ≈ ~$950) |
| NPS | ≥ 40 |

## 9. 关键风险与应对

| 风险 | 概率 | 影响 | 应对 |
|---|---|---|---|
| **B2D 调性输出质量不达标** | 中 | 高(核心壁垒) | 早期人工审核 100% 输出,建立 prompt corpus |
| **Chrome 扩展开发者市场太小** | 低 | 中 | 早做 v2 演进,产品架构预留扩展点 |
| **Pixero / PH GPT 进入正面赛道** | 低 | 中 | 持续监控,凭借垂直 + 调性 + 多渠道防守 |
| **CWS / Reddit / PH 数据抓取被限流** | 中 | 中 | 多源数据 + 降频 + 缓存 + 公开 API 优先 |
| **LLM 成本超预期** | 中 | 高 | 模型路由(简单任务用便宜模型)+ 缓存 + Free 层限流 |
| **PMF 验证失败** | 中 | 高 | 6 周 MVP 后强制 PMF check,LDR < 15% 则重新评估 |
